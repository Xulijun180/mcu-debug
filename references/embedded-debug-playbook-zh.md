## 一、总体原则

嵌入式系统的 bug 往往表现和根因之间隔了好几层，比如 `printf` 崩溃的根因可能是 DMA 溢出。调试的核心思路是：从现象出发，逐层缩小范围，最终定位到具体的硬件或软件行为。

三条基本准则：

- 先观察，后动手。搞清楚现象的全貌再开始改代码，避免盲目试错。
- 一次只改一个变量。同时改两个东西，你永远不知道是哪个起了作用。
- 相信硬件在说话。示波器、逻辑分析仪、调试器看到的才是事实，不要靠猜。

## 二、启动阶段问题

### 1. 程序烧录后不运行

现象：烧录成功但芯片无反应，LED 不亮，串口无输出。

排查步骤：

1. 检查 BOOT 引脚配置。STM32 的 BOOT0/BOOT1 决定从哪里启动。BOOT0=1 会从 System Memory 启动，而不是用户 Flash。
2. 检查复位是否释放。有些调试器烧录后不自动复位，可以手动按一下 RESET。
3. 检查时钟配置是否失败。如果 HSE 晶振没焊好或频率不对，`SystemClock_Config()` 可能会卡在等待 HSE Ready。
4. 检查 Flash 地址是否正确。查看链接脚本中 FLASH 起始地址是否和芯片一致；有 bootloader 时通常需要偏移。
5. 看调试器能否连接。如果 SWD 都连不上，可能是 SWD 引脚被复用成了 GPIO，需要在 RESET 时按住连接。

### 2. 程序运行但外设初始化失败

现象：卡在 `HAL_Init()` 或某个 `MX_xxx_Init()` 中。

排查步骤：

1. 单步跟踪。用调试器单步执行，看卡在哪一行。
2. 检查 HAL 超时。很多 HAL 函数内部有 `while` 循环等待标志位，超时说明硬件没响应。
3. 检查初始化顺序。某些外设有依赖关系，比如 DMA 时钟必须在使用 DMA 的外设之前使能。

## 三、运行阶段崩溃类问题

### 1. HardFault

现象：程序卡死，调试器显示 PC 在 `HardFault_Handler` 的 `while(1)` 中。

排查步骤：

1. 读 fault 状态寄存器：
   - `SCB->CFSR`：综合故障状态。
   - `SCB->CFSR & 0xFF`：MemManage Fault。
   - `SCB->CFSR & 0xFF00`：Bus Fault。
   - `SCB->CFSR & 0xFFFF0000`：Usage Fault。
2. 读故障地址：
   - `SCB->BFAR`：Bus Fault 地址。
   - `SCB->MMFAR`：MemManage Fault 地址。
3. 恢复调用栈：
   - 进入 HardFault 时，CPU 会自动将 R0-R3、R12、LR、PC、xPSR 压栈。
   - 从 SP 指向的位置读出这些寄存器，其中 PC 就是出错指令地址。
   - 用 `addr2line` 或 map 文件把地址转换为源码行号。
4. 对照常见原因：

| CFSR 标志位 | 含义 | 常见原因 |
|------|------|------|
| `IACCVIOL` | 指令访问违规 | 函数指针被破坏，跳转到非法地址 |
| `DACCVIOL` | 数据访问违规 | 空指针解引用、数组越界 |
| `MUNSTKERR` | 出栈时 MemManage | 栈指针被破坏 |
| `PRECISERR` | 精确 Bus Fault | 访问了未映射地址 |
| `IMPRECISERR` | 非精确 Bus Fault | 写缓冲区中的写操作失败 |
| `UNDEFINSTR` | 未定义指令 | PC 跑飞到数据区，或 Thumb/ARM 模式混乱 |
| `UNALIGNED` | 未对齐访问 | Cortex-M0 不支持非对齐，M3/M4 可配置 |
| `DIVBYZERO` | 除零 | 需要在 `SCB->CCR` 中使能 `DIV_0_TRP` |

### 2. 栈溢出

现象：程序随机崩溃，HardFault 的出错地址每次不同，或者全局变量莫名被改。

排查步骤：

1. 检查栈大小。查看链接脚本中 `_Min_Stack_Size` 是否足够。
2. 检查局部变量。函数中是否有大数组分配在栈上，例如 `char buf[1024]`。
3. 使用栈涂色法检测：
   - 在 startup 代码中用 `0xDEADBEEF` 填充整个栈区。
   - 运行一段时间后用调试器查看栈区，从栈底向上找到第一个不是 `0xDEADBEEF` 的位置。
   - 这就是实际最大栈使用深度。
4. 检查是否有意外递归。
5. 检查中断中的栈使用。Cortex-M 的中断共享 MSP，嵌套中断会叠加消耗栈空间。

### 3. 程序跑飞

现象：调试器暂停后发现 PC 在一个完全不合理的地址，例如 `0x00000000` 或 `0xFFFFFFFE`。

排查步骤：

1. 检查函数指针是否被破坏。
2. 检查中断向量表是否被覆盖。如果 DMA 或数组越界写到了 RAM 开头，而向量表又放在 RAM，中断触发时会跳到垃圾地址。
3. 检查返回地址是否被破坏。栈溢出可能覆盖函数返回地址。
4. 看 LR 寄存器。如果 LR=`0xFFFFFFF9` 或 `0xFFFFFFFD`，说明可能是在中断返回时出错。

## 四、外设相关问题

### 1. GPIO

常见问题：

- 输出没反应：时钟没开、模式配错、引脚被其他外设复用。
- 输入读不到：上下拉配置不对、外部电路没有驱动、引脚被锁定。
- 翻转速度不够：GPIO Speed 等级太低，高频翻转时波形失真。

调试技巧：

- 用万用表直接量引脚电压，先区分软件问题和硬件问题。

### 2. UART / 串口

常见问题：

- 完全没输出：TX/RX 接反、波特率不匹配、时钟源频率错误。
- 输出乱码：波特率偏差过大、数据位/停止位/校验位不一致。
- 能发不能收：RX 中断没使能、RXNE 标志没清。
- DMA 发送只发了一次：用了 Normal 模式，发完一次后没有重新启动。

调试技巧：

- 用逻辑分析仪抓 TX 波形，解码实际波特率和数据内容。
- 先用最简单的轮询发送确认硬件通路正常，再切换到中断或 DMA。

### 3. ADC

常见问题：

- 读出来全是 0 或 4095：引脚没配成模拟模式、VREF+ 没接、采样时间太短。
- 数据跳动大：采样时间不够、电源噪声大、缺少滤波。
- DMA 传输数据错位：多通道扫描时 DMA buffer 顺序和通道配置顺序不一致。
- 连续转换 + DMA 溢出：DMA 传输数量超过 buffer 大小。

调试技巧：

- 先把 ADC 引脚接到一个已知电压，比如 `VCC/2`。
- 先跑通单通道轮询，再加多通道、DMA、连续转换。

### 4. I2C

常见问题：

- 通信超时：上拉电阻没接或阻值不对、从设备地址错误。
- 总线锁死：从设备中途复位，SDA 被拉低不释放。
- ACK/NACK 错误：地址不对、从设备没上电、速率太高。
- 某些 STM32 型号 I2C 外设有已知硬件 bug。

调试技巧：

- 用逻辑分析仪抓 SDA/SCL 波形，看地址和 ACK。
- 先用 100kHz 标准模式调通，再提高速率。

### 5. SPI

常见问题：

- MISO 全是 `0xFF` 或 `0x00`：CS 没拉低、CPOL/CPHA 不匹配、从设备没上电。
- 数据偏移：MSB/LSB 顺序配反、位宽不对。
- DMA 传输异常：SPI DMA 往往需要同时处理 TX 和 RX。
- 速率过高：超过从设备最大支持频率。

调试技巧：

- 先发一个已知字节，例如 `0xAA`。
- 确认 CS、SCLK、MOSI、MISO 四根线都接对。

### 6. 定时器

常见问题：

- 中断频率不对：预分频器和自动重装值计算错误。
- PWM 没输出：没调用 `HAL_TIM_PWM_Start()`、输出比较模式配错、GPIO 没设成 AF。
- 更新中断不触发：`HAL_TIM_Base_Start_IT()` 没调用、NVIC 没使能。
- ARR 影子寄存器导致修改后不立刻生效。

### 7. DMA

常见问题：

- 传输不启动：DMA 时钟没开、通道/流选择错误。
- 数据错乱：数据宽度不匹配、内存地址增量没开。
- 溢出覆盖：传输数量大于 buffer 大小，越界写入相邻内存。
- Circular 模式下的竞态：CPU 还没处理完数据，DMA 已经开始覆盖。
- 传输完成后不能重新启动：Normal 模式下 NDTR 归零后没有正确 Stop/Start。

调试技巧：

- 用调试器读 DMA 的 NDTR 寄存器，确认传输是否在进行。
- 在 DMA 完成回调中翻转 GPIO，用示波器确认传输时机。

## 五、内存相关问题

### 1. 堆溢出 / malloc 失败

现象：`malloc` 返回 `NULL`、`printf("%.3f")` 触发断言、程序运行一段时间后崩溃。

排查步骤：

1. 检查堆大小。STM32CubeIDE 默认可能只给 `0x200`。
2. 在 `_sbrk()` 中加高水位标记，统计堆最大使用量。
3. 检查是否有内存泄漏。
4. 注意谁在偷偷用堆：`printf`、`sprintf`、`sscanf`、`_dtoa_r` 等都会走堆分配。

### 2. 栈和堆碰撞

现象：程序运行一段时间后行为异常，堆分配和栈使用都接近极限。

典型布局：

```text
┌─────────────┐ 高地址
│    栈        │ ← SP 向下增长
│    ↓        │
│   (空闲)     │
│    ↑        │
│    堆        │ ← sbrk 向上增长
├─────────────┤
│   .bss      │
│   .data     │
└─────────────┘ 低地址
```

预防措施：

- 在 `_sbrk()` 中检查堆是否碰到栈。
- 使用 MPU 设置 guard region。
- 尽量减少动态分配。

### 3. 内存被意外覆盖

现象：某个全局变量的值莫名其妙变了，但代码里没有任何地方写它。

排查步骤：

1. 对变量地址设 data watchpoint。
2. 查 map 文件看它的邻居。
3. 可以优先检查这些常见的“隐形写手”：
   - DMA 溢出
   - 栈溢出
   - `memcpy` / `memset` 长度参数错误
   - 野指针

## 六、中断相关问题

### 1. 中断不触发

排查清单：

1. 全局中断是否开启。
2. NVIC 是否使能。
3. 外设中断是否使能。
4. 中断优先级是否被屏蔽。
5. 中断处理函数名是否和向量表一致。

### 2. 中断触发但行为异常

常见问题：

- 标志位没清除，导致中断淹没 CPU。
- 中断中做了太多事。
- 中断优先级反转。
- 共享变量没保护。

### 3. 中断优先级配置

Cortex-M 规则：

- 数值越小优先级越高。
- 抢占优先级决定能否打断当前中断。
- 子优先级决定同时挂起时谁先执行。

常见错误：

- SysTick 优先级过低，导致 `HAL_Delay()` 卡死。
- DMA 完成中断优先级低于消费 DMA 数据的中断。

## 七、时钟与电源问题

### 1. 时钟配置错误

现象：串口乱码、定时器频率偏差、ADC 采样率不对。

排查步骤：

1. 检查时钟树。
2. 检查 `HSE_VALUE` 是否和实际晶振一致。
3. 检查 APB 分频和定时器倍频关系。
4. 必要时用 MCO 输出系统时钟，用示波器实测。

### 2. 电源问题

现象：芯片工作不稳定、ADC 数据跳动大、上电后不运行。

排查步骤：

1. 用万用表测 VDD。
2. 检查去耦电容。
3. 用示波器看电源纹波。

## 八、数据与通信问题

### 1. 数据异常的通用排查流程

```text
数据不对
  → 发送端发的对不对？
    → 对 → 线上波形对不对？
      → 对 → 接收端收的对不对？
        → 不对 → 接收端解析逻辑有问题
      → 不对 → 硬件信号问题
    → 不对 → 发送端组包逻辑有问题
```

### 2. 常见数据问题

- 字节序
- 结构体对齐
- `volatile` 缺失
- 竞态条件
- 类型转换错误

## 九、调试工具与技巧

### 1. 调试器

| 功能 | 用途 |
|------|------|
| 断点 | 在指定代码行暂停，检查变量值 |
| 数据断点 | 监控内存地址的读写 |
| 寄存器查看 | 直接读外设寄存器 |
| 实时变量查看 | 不暂停程序查看变量 |
| SWO / ITM 跟踪 | 通过 SWD 输出调试信息 |

### 2. 串口打印

- 最简单直接的调试手段。
- `printf` 本身可能需要堆分配，尤其 `%f`。
- 在 fault handler 中打印时，不要用 `printf`，直接操作 UART 数据寄存器。

### 3. 逻辑分析仪 / 示波器

- 看通信波形是否符合预期。
- 用 GPIO 翻转标记执行时间点。
- 测量中断响应延迟和 DMA 传输时机。

### 4. Map 文件与 `nm`

```bash
arm-none-eabi-nm -n firmware.elf | grep "^2000"
```

当你知道某个地址被破坏时，查 map 文件可以定位变量和它的邻居。

### 5. 栈涂色法

启动时用 `0xDEADBEEF` 填充栈区，运行后检查覆盖范围。

### 6. 编译器辅助选项

| 选项 | 作用 |
|------|------|
| `-fstack-usage` | 生成每个函数的栈使用量报告 |
| `-Wstack-usage=N` | 栈使用超过 N 字节时警告 |
| `-Og` | 更适合调试的优化级别 |

## 十、代表性案例：DMA 溢出导致 printf 断言失败

### 现象

串口输出：

```text
Assertion "Balloc succeeded" failed: file "mprec.c", line 778
```

### 表面分析

`Balloc` 是 newlib 内部为 `printf("%.3f")` 做浮点格式化时的分配函数。断言失败说明堆分配失败。

### 深入排查

1. 堆大小理论上够用。
2. 纯字符串 `printf("hello")` 正常，只有浮点格式化会崩。
3. 对 `__sbrk_heap_end` 设 data watchpoint，发现被 DMA 覆盖。
4. 检查 DMA 配置时发现传输数量远大于 buffer 大小。

### 根因

DMA 传输数量远大于 buffer 大小，溢出的数据覆盖了堆管理指针 `__sbrk_heap_end`，导致后续 `_sbrk()` 失败，malloc 返回 `NULL`，最终 `_dtoa_r` 里的断言触发。

### 完整调用链

```text
printf("%.3fV", voltage)
  → _vfprintf_r()
    → _dtoa_r()
      → _Balloc()
        → _calloc_r()
          → _malloc_r()
            → _sbrk()
              → return (void*)-1
          → return NULL
        → return NULL
      → assert("Balloc succeeded")
        → __assert_func()
          → fprintf(stderr, "Assertion ... failed ...")
            → UART 输出
```

### 教训

- DMA 不会主动替你检查应用层设定的边界。
- 当软件层看起来无明显原因地崩溃时，可以优先检查 DMA、栈和堆这类容易破坏内存的因素。
- 表现和根因之间可能隔着多层调用链，需要结合现场逐层回溯。
- data watchpoint 往往是定位内存破坏问题的高价值工具之一。

## 十一、排查心法

1. 先复现，再分析。
2. 用二分法缩小范围。
3. 不要只盯着软件逻辑，硬件、DMA、中断和电源问题也经常参与形成根因。
4. 善用 watchpoint。
5. 看 map 文件。
6. 从已知到未知。
7. 记录试过的东西。
8. 怀疑一切默认值。
